## Описание базы данных

Эта база данных моделирует бизнес‑процессы небольшой торговой компании (по аналогии с классической базой Northwind). Она содержит таблицы для управления клиентами, продуктами, заказами, сотрудниками, логистикой и демографическими сегментами. Основная цель — учёт операций по продажам, управлению запасами, взаимодействию с поставщиками и клиентами, а также логистике доставки.

### Ключевые сущности и их назначение

- **Клиенты (`customers`)**: хранит информацию о клиентах (название компании, контакты, адрес, телефон). Используется для отслеживания заказов и взаимодействия с клиентами.
- **Заказы (`orders`)**: фиксирует детали транзакций (кто заказал, кто обработал, даты, способ доставки, стоимость). Связан с клиентами и сотрудниками.
- **Продукты (`products`)**: содержит данные о товарах (название, поставщик, категория, цена, остатки на складе). Связан с категориями и поставщиками.
- **Категории (`categories`)**: группирует продукты по типам (например, напитки, кондитерские изделия). Используется для категоризации товаров.
- **Поставщики (`suppliers`)**: хранит данные о поставщиках (контакты, адрес, сайт). Связан с продуктами.
- **Сотрудники (`employees`)**: информация о персонале (ФИО, должность, контакты, фото). Отвечает за обработку заказов и управление территориями.
- **Службы доставки (`shippers`)**: список компаний, осуществляющих логистику. Связан с заказами (поле `ship_via`).
- **Демография клиентов (`customer_demographics` + `customer_customer_demo`)**: позволяет сегментировать клиентов по типам (B2B, частные лица и т. д.). Реализует связь многие‑ко‑многим через промежуточную таблицу.
- **Территории и регионы (`territories`, `region`)**: географическая привязка (регион → территория). Используется для распределения зон ответственности сотрудников.
- **Связь сотрудников и территорий (`employee_territories`)**: промежуточная таблица для реализации связи многие‑ко‑многим (один сотрудник может обслуживать несколько территорий, одна территория — нескольких сотрудников).
- **Детали заказа (`order_details`)**: конкретика по товарам в заказе (цена, количество, скидки). Связан с заказами и продуктами.
- **Штаты США (`us_states`)**: вспомогательная таблица (возможно, для гео‑анализа или фильтрации по регионам).

---

### Основные бизнес‑процессы, которые отражает БД

1. **Управление продуктами**: добавление, обновление, удаление товаров, отслеживание запасов (через `products`, `categories`, `suppliers`).
2. **Обработка заказов**: регистрация заказа (`orders`), подбор товаров (`order_details`), назначение ответственного сотрудника (`employee_id`), выбор службы доставки (`ship_via`), фиксация адресов доставки.
3. **Работа с клиентами**: учёт контактов и адресов клиентов (`customers`), сегментация по демографическим типам (`customer_customer_demo`, `customer_demographics`).
4. **Логистика**: выбор службы доставки из списка (`shippers`), учёт адресов доставки в заказах.
5. **Управление персоналом**: хранение данных о сотрудниках (`employees`), распределение зон ответственности через территории (`employee_territories`, `territories`).
6. **Аналитика и отчётность**: возможность группировать данные по категориям, регионам, поставщикам, типам клиентов.

---

### Особенности структуры

- **Связи многие‑ко‑многим**: реализованы через промежуточные таблицы (`customer_customer_demo`, `employee_territories`), что позволяет гибко управлять сложными отношениями.
- **Типы данных**: использование `bpchar` (фиксированная длина) для идентификаторов (`customer_id`, `customer_type_id`) — возможно, для стандартизации и оптимизации хранения.
- **Медиа‑данные**: поля `bytea` (`picture`, `photo`) и `photo_path` для хранения изображений (например, логотипов категорий или фото сотрудников).
- **Гибкость демографии**: разделение на `customer_demographics` (описание типов) и `customer_customer_demo` (привязка клиентов к типам) даёт возможность легко добавлять новые сегменты клиентов.

### Пример сценария работы

Клиент делает заказ:
1. Данные клиента (`customers`) привязываются к заказу (`orders`).
2. Сотрудник (`employees`) обрабатывает заказ.
3. Товары из заказа (`order_details`) связаны с `products` (и, через них, с `categories` и `suppliers`).
4. Для доставки выбирается служба из `shippers` (поле `ship_via` в `orders`).
5. Если требуется гео‑анализ, можно использовать `territories` и `region` для привязки к зонам ответственности сотрудников.

### Основные связи (ER-диаграмма)

1. **categories** → **products**
   - Внешний ключ: `category_id` в `products` ссылается на `category_id` в `categories`.
   - Связь: один ко многим (одна категория содержит много продуктов).

2. **suppliers** → **products**
   - Внешний ключ: `supplier_id` в `products` ссылается на `supplier_id` в `suppliers`.
   - Связь: один ко многим (один поставщик поставляет много продуктов).

3. **customers** → **orders**
   - Внешний ключ: `customer_id` в `orders` ссылается на `customer_id` в `customers`.
   - Связь: один ко многим (один клиент делает много заказов).

4. **employees** → **orders**
   - Внешний ключ: `employee_id` в `orders` ссылается на `employee_id` в `employees`.
   - Связь: один ко многим (один сотрудник обрабатывает много заказов).

5. **shippers** → **orders**
   - Внешний ключ: `ship_via` в `orders` ссылается на `shipper_id` в `shippers`.
   - Связь: один ко многим (одна служба доставки участвует в многих заказах).

6. **region** → **territories**
   - Внешний ключ: `region_id` в `territories` ссылается на `region_id` в `region`.
   - Связь: один ко многим (одна область включает много территорий).

7. **territories** → **employee_territories**
   - Внешний ключ: `territory_id` в `employee_territories` ссылается на `territory_id` в `territories`.
   - Связь: многие ко многим через промежуточную таблицу (один сотрудник может обслуживать много территорий, одна территория обслуживается многими сотрудниками).

8. **employees** → **employee_territories**
   - Внешний ключ: `employee_id` в `employee_territories` ссылается на `employee_id` в `employees`.
   - Связь: многие ко многим через промежуточную таблицу.

9. **customer_demographics** → **customer_customer_demo**
   - Внешний ключ: `customer_type_id` в `customer_customer_demo` ссылается на `customer_type_id` в `customer_demographics`.
   - Связь: один ко многим (один тип клиента может быть связан с множеством клиентов).

10. **customers** → **customer_customer_demo**
   - Внешний ключ: `customer_id` в `customer_customer_demo` ссылается на `customer_id` в `customers`.
   - Связь: один ко многим (один клиент может иметь много типов демографии).
